# Spec v0.1 — Dogfood Feedback

**Build target:** `smartchip.govtech.qr-offline-authorization @ 1.0.0`
**Outcome:** Built, signed, verified end-to-end. Tampering detected correctly.
**Crypto:** Real Ed25519 keypair; 64-byte signatures; 32-byte keys; 11 files tracked.
**Scripts:** `build.py` (publisher side), `verify.py` (consumer side).

Running the spec as an actual implementation surfaced 12 issues — some
minor, some genuinely load-bearing. Each is tagged by severity.

---

## A. Load-bearing gaps (must fix before v0.2)

### A1. Attestations are not bound to content. `[severity: high]`

Current spec: attestations name the pack by `pack_id@version`. An attacker
with a compromised publisher key can sign a valid-looking attestation for
version `1.0.0` and ship it alongside arbitrary content that happens to
also claim version `1.0.0`. The only thing binding attestation to content
is the social claim inside `pack_id@version`.

**Fix.** Every attestation must include the `merkle_root` field. Consumer
verifies the attestation is bound to the specific content it came with.

```json
{
  "spec": "autoevolve-attestation/redaction/0.1",
  "pack": "smartchip.govtech.qr-offline-authorization@1.0.0",
  "merkle_root": "sha256:dff97c99...",   // ← add this
  ...
}
```

This means attestations are built AFTER the lock is computed, which is a
pipeline ordering change in the distiller.

### A2. Manifest ↔ lock_hash circular dependency. `[severity: medium]`

Spec §3 shows `lock_hash` as a manifest field. But `pack.lock` contains
the hash of `pack.manifest.yaml`. So computing the manifest's `lock_hash`
requires the lock, which requires the manifest hash, which depends on its
content including `lock_hash`. Circular.

My build silently omitted `lock_hash` from the manifest. The signature
over `pack.lock` is what binds content anyway, so `lock_hash` in manifest
is redundant.

**Fix.** Remove `lock_hash` from manifest. Document that binding comes via
publisher signature over `pack.lock` and that the manifest is itself one
of the files listed in the lock.

### A3. DID resolution / trust bootstrapping undefined. `[severity: high]`

Spec §6 step 3e says "verify publisher.pubkey matches publisher.id (DID
resolution or known-key list)". But:
- How does the consumer build the known-key list the first time?
- If consumer does online DID resolution (`did:web:` → HTTPS fetch), does
  that defeat the offline-verifiable promise?
- What if DID document is unreachable?

My implementation used a hardcoded dict — fine for demo, not for a
marketplace.

**Fix.** Spec should declare the trust bootstrapping model explicitly.
Options to evaluate:
- **TOFU (Trust On First Use)**: first time a publisher is seen, record
  their key; warn if it changes.
- **Allowlist**: consumer's policy engine contains a static list of
  trusted publisher keys; unknown publishers auto-rejected.
- **Registry-assisted**: registry publishes a signed directory of
  publisher keys; consumer trusts registry's meta-key; registry trust
  is bootstrapped out-of-band.

Recommendation: allowlist as default for v0.2; TOFU as opt-in; registry-
assisted deferred.

### A4. No timestamping authority. `[severity: medium]`

Attestations have a `build_timestamp` but it's publisher-claimed. If a
publisher's key is compromised at time T, an attacker can retroactively
sign attestations with timestamps < T. Consumer cannot detect.

**Fix.** v0.2: require RFC 3161 timestamp from an external authority on
every attestation. For v0.1, acknowledge this limit explicitly.

### A5. Key rotation has no path. `[severity: medium]`

When publisher rotates keys, old attestations signed with old keys become
unverifiable (old key may be gone from DID doc). Current attestations have
no `key_id` field.

**Fix.** Every signature field becomes:
```json
"signature": {
  "algorithm": "ed25519",
  "key_id": "smartchip-key-2026Q2",
  "value": "hex..."
}
```
and publisher maintains a historical key registry keyed by `key_id`.

## B. Clarity gaps (spec should say explicitly)

### B1. Canonical JSON form ambiguous. `[severity: low]`

Spec says "RFC 8785 (canonical JSON)". My build used a simpler form:
`json.dumps(sort_keys=True, separators=(",",":"), ensure_ascii=False)`.
Real RFC 8785 additionally requires specific Unicode normalization and
number serialization rules.

**Fix.** Either (a) commit to full RFC 8785 and require a validated
library, or (b) define a "canonical-subset-2026-04" that's simpler and
document it precisely. For v0.1, option (b).

### B2. Signature covers what exactly? `[severity: low]`

In my implementation: signature covers `canonical_json(obj minus "signature" field)`.
Spec doesn't say this. Could be interpreted as "sign the whole document,
then append signature field" which would not round-trip-verify.

**Fix.** Spec explicitly: "signature is computed over the canonical-JSON
serialization of the attestation with the `signature` field removed".

### B3. What files go into pack.lock? `[severity: low]`

Spec §4 lists a sample lock. My implementation included all files except
`pack.lock` itself and `signatures/*`. That's an inferred rule.

**Fix.** Spec says literally: "pack.lock covers all files under the pack
root recursively, except (a) pack.lock itself, (b) anything under
`signatures/`".

### B4. Page file types not bounded. `[severity: low]`

Spec says "pages/ directory contains markdown files". But what about
diagrams (SVG), test fixtures (JSON), images referenced in markdown?
My build used only .md but real packs will include other assets.

**Fix.** Spec: "pages/ may contain markdown (.md) and supporting assets.
Asset types permitted: svg, png, jpg, json, yaml. All listed in pack.lock."

### B5. Duplication between manifest.license and license.json attestation. `[severity: low]`

Same info appears in two places. If they disagree, which wins?

**Fix.** Declare `license.json` as source of truth. Manifest block is a
convenience summary; on mismatch, pack is rejected.

## C. Missing fields surfaced during use

### C1. `residual_risk_notes` in redaction.json. `[severity: low]`

While writing the redaction attestation for this pack I found myself
wanting to note "consumers who know the set of national QR border
systems can probably guess the client from the crosswalk hint". The
redaction spec v0.1 §8 mentioned `residual_risk` as a future field.
Dogfood confirmed: it's needed now, not later.

**Fix.** Add `residual_risk_notes: [string]` as required field in
redaction attestation v0.1.1. Forces publisher to articulate what their
redaction does NOT protect against.

### C2. Pack needs a canonical URL. `[severity: low]`

Nothing in the manifest points at "where can I fetch this pack's tarball".
Registry is expected to answer this, but for direct sharing (email,
Slack, etc.) a URL in the manifest would help.

**Fix.** Add optional `canonical_url` to manifest: `"https://smartchip.ie/packs/smartchip.govtech.qr-offline-authorization-1.0.0.tar"`.

### C3. Dependencies need registry hints. `[severity: low]`

Manifest lists `dependencies: [pack_id, version]`. My verify script
showed "would recursively verify..." but where from? Multi-registry
world needs dependencies to optionally say *which registry* the dep
comes from.

**Fix.** Extend dependency entry:
```yaml
dependencies:
  - pack_id: "base.crypto.ed25519-signing"
    version: "^1.0"
    scope: "references"
    registry_hint: "https://registry.autoevolve.io"   # optional
```

## D. What worked well (preserve these)

1. **Two-level signing** (content Merkle → lock → publisher.sig). Clean
   separation; 30 lines of verification code as promised; tampering
   detected immediately and at the right layer.

2. **Fail-closed policy_surface.** Pack declares which dimensions can be
   evaluated; anything consumer requires beyond that list rejects the
   pack. Prevents silent policy degradation.

3. **Separate-file attestations.** Being able to sign and verify each
   attestation independently makes third-party attestations (e.g., an
   audit firm signing its own `evaluation.json` alongside the publisher's)
   natural. This is the right primitive for a marketplace.

4. **Pattern-level granularity.** 13 KB of content across 5 pages felt
   like the right size: complete enough to be useful, small enough to
   version precisely, composable enough to be combined via dependencies.

5. **Crosswalk with explicit equivalence assertions.** Turned out easy
   to write and forces publisher to declare interoperability claims
   explicitly rather than implicitly.
