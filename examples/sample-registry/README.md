# examples/sample-registry/

A reference layout for a GitHub-hosted kb-registry. Fork or template
this directory to stand up your own.

## Layout

```
sample-registry/
├── publishers/
│   └── {did-safe}/
│       └── keys.json          Trust anchor: publisher's active keys
├── packs/
│   └── {pack_id}/
│       └── {version}.tar      Signed pack tarballs (LFS-friendly)
├── index.json                 Regenerated by CI on every merge
├── scripts/
│   └── verify_submission.py   Standalone verifier CI runs pre-merge
├── .github/
│   └── workflows/
│       └── verify-pack.yml    Pre-merge verification workflow
└── README.md                  This file
```

## Consumer side

Point an existing KB at this registry with:

```bash
kb-mcp --root my-kb
# Agent-side:
#   kb/registry_describe/0.1  {"registry_url": "file:///path/to/sample-registry"}
#   kb/registry_search/0.1    {"registry_urls": ["file://..."], "query": "sanctions"}
#   kb/subscribe/0.1          {"registry_url": "file://...", "pack_id": "x", "constraint": "^1.0"}
```

Phase 3 v1 only supports `file://` and bare paths. `https://` and
`git+https://` transports are specified by the AutoEvolve registry
spec but not yet shipped in the reference implementation.

## Publisher side

Two steps:

1. Add (or update) `publishers/<did-safe>/keys.json` listing your
   Ed25519 public keys.
2. Build a signed pack (`kb/publish/0.1` in a local KB) and add the
   tarball to `packs/<pack_id>/<version>.tar`. Open a PR.

The CI workflow in `.github/workflows/verify-pack.yml` runs the
reference `verify_pack` against every new or changed tarball before
the PR can be merged.

## Trust model

The registry is **not** a trust root. Consumers verify each pack
locally against the publisher's keys from `publishers/<did-safe>/keys.json`.
A malicious or buggy registry can hide packs, serve the wrong bytes,
or redirect to wrong tarballs — all of these are caught at the
consumer's verify step (hash mismatch, signature failure), not by
trusting the registry.
