## 2. Architectural shape of a kb-registry

```
registry-root/
тФЬтФАтФА publishers/<did-safe>/keys.json
тФЬтФАтФА packs/<pack_id>/<version>.tar
тФФтФАтФА index.json                       # denormalized; rebuilt from filesystem
```

`index.json` is a **cache**, not a trust root. The registry is the
filesystem; the index exists only so consumers don't have to walk
every directory to discover packs. A consumer that receives a bad
index either sees a stale view (acceptable) or a lie about pack
contents (caught at the consumer's seven-step verify тАФ Phase 2
cryptography is the actual trust root).

Phase 3 v1 accepts `file://` and bare-path URL schemes. The
`Registry` abstraction is deliberately thin so `https://` and
`git+https://` can land as subclasses without disturbing the
contract. See the `unsupported scheme` branch in
`kb_registry.registry._resolve_registry_root` тАФ the extension point.

## 3. D1-D6 adversarial results

| # | Scenario                                      | Outcome | Step                          |
|---|-----------------------------------------------|---------|-------------------------------|
| D1 | Happy path: both publishers trusted          | тЬУ accept | S7 (verified)                |
| D2 | Dependency missing from registry             | тЬУ reject | `dep_resolve_failed`         |
| D3 | Version constraint `^5.0` vs `1.0.0` only    | тЬУ reject | `dep_resolve_failed`         |
| D4 | Content tampering inside the dependency      | тЬУ reject | S2 recursive + breadcrumb    |
| D5a | Strict mode + untrusted dep publisher       | тЬУ reject | `untrusted_dep_publisher`    |
| D5b | inherit-from-parent accepts dep             | тЬУ accept | S7                           |
| D5c | namespace-scoped allow                      | тЬУ accept | S7                           |
| D5d | namespace-scoped deny                       | тЬУ reject | `namespace_publisher_rejected` |
| D6 | Cycle between two packs                      | тЬУ reject | `cycle` or S2 (either valid)   |

D5 quartet is the point: each of the four settings produces the
expected decision on the same registry state. Strict is the v0.1.1
default; inherit-from-parent and namespace-scoped are the two modes
`reports/03-dep-chain-report.md` queued for v0.1.2. Both ship here,
with `max_inherit_depth` honored so a deep malicious chain cannot
auto-trust arbitrary publishers.

D4 validates that the breadcrumb mechanism works: the rejection
message contains `base.crypto@1.0.0` inside the failing path, so a
consumer with a ten-deep chain sees exactly where it broke without
digging through debug logs.

## 4. Cross-registry subscribe round-trip

`tests/test_mcp_registry_tools.py::test_subscribe_via_registry_url`
drives the full flow:

1. Two versions of `demo.reg` published into a file-based registry
   via the real draft/distill/publish pipeline.
2. `index.json` regenerated by `write_index`.
3. Consumer KB calls `kb/subscribe/0.1` with
   `{"registry_url": "file://...", "pack_id": "demo.reg",
   "constraint": "^1.0"}`.
4. Tool fetches the 1.2.0 tarball into a tempdir, runs verify (TOFU
   registers the publisher's key into `.kb/trust-store.yaml`), and
   installs under `subscriptions/did-web-pub.example/demo.reg/1.2.0/`.
5. Tempdirs are cleaned up on both success and failure paths.

## 5. Findings

### 5.1 `inherit-from-parent` bounds are load-bearing

Without `max_inherit_depth`, a single subscribed pack with a long
chain of cross-publisher deps could auto-trust dozens of
publishers. `max_inherit_depth=2` (the default) is aggressive
enough to catch realistic marketplaces тАФ you trust your direct
publisher, you trust one layer below them, and anything deeper
needs explicit operator review. Documented in the redaction
attestation already; now enforced in the recursive verifier.

### 5.2 Namespace globs are practically tree-structured

`consumer.namespace_publishers` uses `fnmatch` globs. Real
taxonomies look like `"base.*"`, `"regulatory.eu.*"`. The current
matcher handles the common cases; a future phase could adopt the
`glob.glob`-style `**` separator for arbitrary depth, but the
single-star form is enough for v0.1.2 semantics.

### 5.3 Recursive tampering still fails at S2, not a dedicated step

D4 predicted S3a per `reports/03-dep-chain-report.md ┬з2`. In
practice the content_root recomputation in S2 fires first for an
in-dep page edit, exactly as the Phase 2 defense-in-depth finding
in `reports/02` predicted. This is correct; recording it here so
future readers don't chase a step-number discrepancy.

### 5.4 Cycle detection remains defense-in-depth, not primary

D6 almost never surfaces the `cycle` step because two legitimately
rebuilt packs that cycle would both have fresh content_roots that
differ from whatever is recorded in their cross-references'
attestations. `report/03 ┬з4.3` predicted this; our test accepts
either rejection signal (`cycle` OR `S2` OR `S3a`). The cycle
guard stays in place because the one configuration that reaches
it тАФ two legitimately-signed packs with mutual static deps and no
content drift тАФ is also the one that would actually benefit from
an explicit diagnostic.

### 5.5 `registry_hint` must be consulted per-dep

The recursive verifier opens a per-dep registry when the dep
carries `registry_hint`. This matters because `base.crypto` might
live in a foundation registry while the consuming `app.billing`
lives on a company registry. Cross-registry deps work today without
any federation protocol between registries тАФ the consumer is the
one that queries them, per the consumer-driven-federation
positioning in the registry spec ┬з7.

## 7. What's explicitly NOT proven

- **No https:// / git+https:// transports.** Only file:// plus bare
  paths. Production registries will want a real network layer; the
  `Registry` abstraction is the extension point.
- **No registry write operations over the wire.** Publishing into
  the sample registry happens via a git PR, not a
  `registry/submit/0.1` MCP tool. This is fine for v1 (the roadmap
  picked git-based registry explicitly), but consortium registries
  that want permissionless submit would need the submit tool.
- **No automatic DID document resolution.** Publisher keys come from
  the registry's `publishers/<did-safe>/keys.json` file. Real
  `did:web:` resolution over HTTPS is spec'd in
  `autoevolve-spec-v0.1.1.md ┬з6.3` but deferred.
- **No pub/sub.** Consumers poll `list_versions` or re-run
  `kb/registry_search/0.1`. Real-time notifications are v0.2 per
  registry spec ┬з8 non-goals.
- **No registry-federation trust protocols.** Consortium registries
  that want to declare which other registries they mirror or trust
  have no protocol hook in v0.1.2.

These are the same honest limits the ROADMAP listed; nothing Phase 3
surfaced altered them.
