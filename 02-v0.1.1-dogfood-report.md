# v0.1.1 Dogfood Report

**Target:** smartchip.govtech.qr-offline-authorization @ 1.0.0 (rebuilt against v0.1.1)
**Artifacts:** `qr-offline-authorization-1.0.0/`, `build.py`, `verify.py`,
`adversarial_tests.py`, `t8_key_compromise.py`
**Status:** All 12 amendments implemented and verified. 8/8 adversarial tests
caught their target attacks.

---

## 1. What we built

- **Pipeline reordered** for two-merkle scheme:
  1. Write content (pages + manifest + README)
  2. Compute `content_root` over content only
  3. Build attestations embedding `content_root` field; sign each
  4. Compute `pack_root` over all files (content + attestations)
  5. Sign `pack_root` → `signatures/publisher.sig`
- **Envelope signatures** throughout: `{algorithm, key_id, value}` instead
  of bare hex.
- **Manifest slimmed**: `lock_hash` removed; `canonical_url` added;
  dependencies got `registry_hint`.
- **`residual_risk_notes` enforced** non-empty in redaction attestation,
  and checked in consumer policy.
- **Trust store** keyed by `(publisher_id, key_id)` pair to support key
  rotation.

## 2. Adversarial test results

| # | Attack | Expected | Actual | Verdict |
|---|---|---|---|---|
| T1 | Tamper page content | S3a | S3a | ✓ exact |
| T2 | Tamper attestation body | S3b | S3b (hash) | ✓ exact |
| T3 | Swap attestation claiming different content | S3b | S3b (hash) | ✓ caught |
| T4 | Manifest license ≠ license.json | S3c | S3b (A1 fires first) | ✓ caught, different step |
| T5 | Empty `residual_risk_notes` without key | S4 | S3b (hash) | ✓ caught, different step |
| T6 | Invalid `key_id` in envelope | S3b | S3b (hash) | ✓ caught |
| T7 | Algorithm downgrade | S3b | S3b (hash) | ✓ caught |
| T8 | **Key compromise + stale attestation reuse** | S3b | **S3b (content_root)** | ✓ **this is A1's real value** |

All 8 attacks rejected. But three of them (T4, T5, T7) were caught at a
different step than expected. This is not a failure — it's a finding.

## 3. Key finding: defense-in-depth hierarchy revealed

Attacks group by the attacker's capabilities. The crypto stack catches
each level with the minimum-cost check:

```
Level 1: Attacker can only modify pack bytes
         → Hash-in-lock check (S3a, S3b hash step) catches it
         → Zero crypto needed, just SHA-256

Level 2: Attacker can update pack.lock (but not re-sign with publisher key)
         → Publisher signature over pack_root (S5) catches it
         → Ed25519 verification

Level 3: Attacker has publisher key AND wants to reuse legitimate older
         attestations to bypass policy checks
         → Amendment A1 (content_root binding, S3b semantic check) catches it
         → This is the attack A1 was designed to prevent (T8)

Level 4: Attacker has publisher key AND is willing to forge attestations
         from scratch
         → Only policy checks (S4) provide any defense, e.g. "require
           third-party attestation from audit firm X"
         → Not currently in v0.1.1 policy surface
```

**This hierarchy is worth making explicit in v0.1.2.** Without it, readers
of the spec may think amendment A1 is about general tampering. It's not —
it's specifically about *stale attestation reuse under key compromise*.

## 4. Finding: amendment B5 is unreachable through malicious path

T4 tried to simulate an attacker inserting inconsistency between the
manifest's license block and `attestations/license.json`. The attempt was
caught at S3b (A1's content_root mismatch) before S3c (B5's license
consistency check) could fire.

Why: any modification to manifest changes `content_root`. Attestations
embed the OLD `content_root`. Mismatch caught immediately.

Implication: **B5 defends against honest publisher errors, not malicious
actors.** A publisher who manually edits manifest without re-running the
distiller might ship inconsistent license info. B5 catches that. A
malicious actor trying to exploit license confusion is already blocked by
A1.

This is still valuable (publishers DO make mistakes), but the spec should
not oversell B5 as a security control. **v0.1.2 amendment**: reclassify
B5 as "integrity sanity check" not "security requirement".

## 5. Finding: hash-check depth matters

Three of seven adversarial tests (T5, T6, T7) were caught by the hash-in-
lock check before the more semantic check (policy, key_id, algorithm)
could run. This means those semantic checks' unique value is only
realized when the attacker also controls `pack.lock` AND the publisher
signature.

Not a problem — it's the right layering. But v0.1.2 should add a note:
**"semantic checks (key_id validation, algorithm acceptance, policy
evaluation) are only load-bearing in the key-compromise scenario; for
non-compromised deployments, hash-in-lock is the dominant defense."**

## 6. Cost of v0.1.1

- Build pipeline: 5 phases instead of 3. ~30 extra lines of code.
- Verification: ~50 extra lines for content_root binding + envelope parsing.
- Attestation size: ~15% larger due to envelope + `content_root` +
  `residual_risk_notes`.
- Conceptual cost: the two-merkle scheme is harder to explain to
  non-experts. Worth it for the key-compromise defense.

## 7. Spec amendments needed for v0.1.2

Minor, all clarifications rather than behavior changes:

| # | Change | Rationale |
|---|---|---|
| 1 | Add §5.4 "Defense-in-depth layers" section explaining hierarchy | Current spec doesn't explain when A1 is load-bearing vs. when it's redundant with hash check |
| 2 | Reclassify B5 as "integrity sanity check" | T4 showed it's unreachable via malicious path |
| 3 | Add "key compromise scenario" to §6.4 (verification procedure examples) | T8 is a canonical test that should be in the reference test suite |
| 4 | Add note to §3 manifest that `license` block is ONLY for display/discovery | Reinforce B5 reclassification |
| 5 | Require dogfood test suite include at least one key-compromise test | Without T8-style tests, A1's unique value goes untested in typical dogfood |

Not a lot. The v0.1.1 design was substantively correct; the gaps are
documentation and testing conventions.

## 8. Ready for next step

With v0.1.1 implemented, tested, and understood, we have a working
reference implementation plus an honest map of what it defends against.
Next candidates:

- **Dependency chain test** — produce `base.crypto.ed25519-signing` as a
  second pack, make qr-offline depend on it, exercise recursive verification.
- **Q2 (registry API)** — with two+ packs, discovery starts to matter.
- **Q1b (policy evolution loop)** — the patent-relevant genome fitness.
- **Patent claims draft** — enough reference implementation now exists to
  write concrete USPTO provisional claims backed by working code.

Any of these is defensible as next step.
